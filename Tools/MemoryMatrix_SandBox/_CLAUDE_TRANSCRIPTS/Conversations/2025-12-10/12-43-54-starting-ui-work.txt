Ahab wants to understand how git tracking and .gitignore work. Let me explain this clearly and concisely.Good question. Here's the mental model:

**Git tracks files in two places:**
1. **Working directory** - actual files on disk
2. **Index/staging area** - what git "knows about" and will include in commits

**The problem:**
When you first `git add .`, git started tracking everything—including `target/` and `node_modules/`. Once tracked, `.gitignore` doesn't help. It only prevents *new* files from being added.

So even after adding `.gitignore`, those 900MB are still in git's index saying "yep, track these."

**The fix:**
```bash
git rm -r --cached target/
```

This does:
- `rm` - remove from git
- `--cached` - only from the index (git's memory), NOT from disk
- `-r` - recursive (whole directory)

So the files stay on your machine, but git "forgets" about them. Next commit reflects that removal. Future `git add .` won't re-add them because `.gitignore` now blocks them.

**TL;DR:**
- `.gitignore` = "don't add new files matching this"
- `git rm --cached` = "forget you ever knew about this file"
- Together = clean repo going forward
