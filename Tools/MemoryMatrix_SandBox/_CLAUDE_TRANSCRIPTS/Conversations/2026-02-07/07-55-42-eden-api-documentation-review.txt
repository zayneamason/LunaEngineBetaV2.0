**What it is:** A creative AI platform with a RESTful API (`api.eden.art`) that wraps generative AI tools (image, video, audio, text) behind an agent/session model. ### Key Patterns

**Session-based communication.** You don't just hit a generation endpoint — you create a session with agent(s), set budgets, then send messages within that session. The agent maintains context across turns. This is their core interaction model.

**Async task polling for generations.** When an agent (or you directly) triggers a tool, it creates a Task. You poll the task for status and results. Standard async job pattern.

**Budget-gated.** Every session has three budget dimensions: manna (credits), tokens (LLM), turns (conversation). This is how they meter usage.

**Multi-agent sessions.** You can put multiple agents in one session with autonomy settings (`auto_reply`, `reply_interval`, `actor_selection_method: "random"`). Agents can talk to each other.

**On-chain agent registry.** This is interesting — agents are ERC-721 NFTs on-chain with IPFS metadata. They have a trainer authorization system. This is the web3/ownership layer.

### Auth

Dead simple: `X-Api-Key` header on every request. No OAuth, no JWT refresh dance.

### What's Interesting for Luna

A few things jump out:

1. **Agent architecture parallel.** Eden's agents have personality + tools + memory + persistent sessions. That's structurally similar to what you're building with Luna. Their session model = your Hub Daemon sessions. Their agent memory = your Memory Matrix. Their tools = your eventual tool integrations.

2. **Integration layer.** They've already solved the "deploy your agent to Discord/Telegram/X" problem. If Luna ever needs social presence, Eden's integration approach (or their actual platform) could be a deployment target.

3. **Generation tools as services.** Rather than running Flux/Kling/ElevenLabs locally, Eden wraps them behind a unified API. For Luna's creative capabilities, this could be a pragmatic "don't build this yourself" layer — let Eden handle the GPU fleet, Luna handles the personality and memory.

4. **The Manna model.** Worth studying for if/when Luna needs to manage resource budgets across tools.

## hello-eden — The Reference Implementation

It's a **Next.js 15 + React 19 + TypeScript + Tailwind** starter app. Standard BFF (backend-for-frontend) pattern — Next.js API routes proxy to Eden so the API key stays server-side. Nothing exotic architecturally.

### What the Client Code Reveals

The `eden.ts` file is the real documentation. Here's what the API surface actually looks like in practice:

**Endpoints confirmed from client code:**

| Endpoint | Method | Purpose |
|----------|--------|---------|
| `/v2/tasks/create` | POST | Submit generation jobs |
| `/v2/tasks/{taskId}` | GET | Poll job status |
| `/v2/sessions` | POST | Create session OR send message (overloaded!) |
| `/v2/sessions/{sessionId}` | GET | Get session state + messages |
| `/v2/agents` | GET | List agents |
| `/v2/agents/{agentId}` | GET | Get agent details |
| `/v2/feed-cursor/creations` | GET | Browse creations (cursor-paginated) |
| `/v2/creations/{creationId}` | GET | Get single creation |

**Interesting design details from the types:**

1. **Agent model is rich.** They've got `persona`, `greeting`, `knowledge`, `voice`, `suggestions`, custom `llm_settings` (model profile, thinking policy — sounds like they're wrapping Claude/GPT with configurable reasoning), `tools` as a boolean map, and `models` as LORA references with `use_when` conditions. That last bit is clever — conditional model selection per context.

2. **Sessions are overloaded.** Same `POST /v2/sessions` endpoint both creates sessions AND sends messages (distinguished by presence of `session_id` in body). A bit smelly but pragmatic.

3. **ToolCall model.** Agents can invoke tools during conversations, which produce `result` arrays with `output` (media files) and `subtool_calls`. This is their function-calling layer — agent decides to use Flux Kontext, that becomes a ToolCall, which produces a Creation.

4. **Budget model is three-dimensional:** `manna_budget` (credits), `token_budget` (LLM tokens), `turn_budget` (conversation turns). Each has a `*_spent` counterpart for tracking.

5. **Autonomy settings.** `auto_reply`, `reply_interval` (ms), `actor_selection_method` ('random' | 'random_exclude_last'). This is their multi-agent orchestration — agents can auto-respond in sessions.

6. **Thinking support.** `thinking` and `thought` fields on messages, plus `thinking_policy` and `thinking_effort_cap` on agent LLM settings. They're exposing extended thinking controls.

1. Luna assembles context — character refs, style ref, narrative position, panel description
