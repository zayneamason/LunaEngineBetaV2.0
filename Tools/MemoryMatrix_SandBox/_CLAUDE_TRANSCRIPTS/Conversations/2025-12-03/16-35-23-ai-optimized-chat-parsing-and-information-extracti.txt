## EXTRACTION TYPES

Recognize and isolate these patterns in conversation:

### FACT
- Claims about the world, codebase, or system state
- Must be falsifiable; has a truth value
- Example: "Memory retrieval takes ~200ms on average"
- **Extract:** exact claim, conditions, any numbers/bounds

### DECISION
- A choice made or affirmed
- Includes: what was chosen, what was rejected, stated rationale
- Example: "We chose Silero VAD ONNX over Whisper-based detection because [reasons]"
- **Extract:** decision, options considered, tradeoffs acknowledged

### PROBLEM
- Something that's broken, unclear, or needs solving
- Can be concrete (bug) or abstract (design gap)
- Example: "Voice interrupt detection is reliable at ~300ms latency but unreliable below that"
- **Extract:** problem statement, reproduction steps if relevant, impact scope

### ASSUMPTION
- Something taken as true without proof
- Often unstated; flag explicitly
- Example: "This assumes JSON serialization preserves insertion order" (false in some Python versions)
- **Extract:** the assumption, why it was made, what breaks if false

### CONNECTION
- Relationship between ideas, components, or decisions
- "X enables Y", "X contradicts Y", "X depends on Y"
- Example: "Smart Turn v3 depends on accurate prosody hints; hints require real-time TTS processing"
- **Extract:** two entities, relationship type, why it matters

### ACTION
- Something that should be done
- Includes: what, who, why, by when (if stated)
- Example: "Rebuild memory indexes to fix retrieval inconsistencies"
- **Extract:** action, owner, priority, blocking dependencies

### OUTCOME
- Result of a prior action, experiment, or decision
- Actual result vs. expected result
- Example: "Token usage dropped from 15k to 2.3k after implementing hot/warm/cold tiers"
- **Extract:** what happened, why it matters, any surprises

## PARSING WORKFLOW

When processing a chat:

1. **Scan for signal markers** (decisions made, problems identified, actions stated, outcomes claimed)
2. **Extract atomically** (one fact/decision/problem per unit, never composite)
3. **Trace relationships** (what depends on what? what contradicts what?)
4. **Assign tags** (domain, subsystem, maturity, urgency)
5. **Flag unknowns** (set `confidence < 100` if not fully verified, add caveats)
6. **Surface conflicts** (if two items contradict, mark `status:DISPUTED`, note in "notes")
7. **Store with metadata** (timestamp, owner, deadline, status)

---

## EFFICIENCY RULES FOR AI PROCESSING

- **Minimize redundancy:** One fact, one entry. No duplication.
- **Explicit relationships:** Don't infer; state `depends_on`, `contradicts`, etc.
- **Strict schema:** No optional fields used arbitrarily; consistency for parsing.
- **Atomic units:** Split composite claims. "We did X and Y because Z" → three entries.
- **Confidence as filter:** Use `confidence` to weight relevance. Low confidence = needs verification.
- **Status as lifecycle:** Track ACTIVE, ARCHIVED, DISPUTED, VERIFIED to manage information churn.
- **Tags as dimensions:** Multiple independent dimensions (domain, subsystem, maturity, urgency) not hierarchy.

---

## WHEN TO EXTRACT vs. IGNORE

**Extract:**
- Decisions made (even tentative)
- Problems named
- Design rationale or tradeoffs explained
- Outcomes with numbers
- Assumptions surfaced
- Actions committed to
- Connections between concepts

**Ignore:**
- Casual observations without context
- Off-topic tangents
- Rhetorical questions
- Repetition of already-extracted items
- Throat-clearing or context-setting fluff

**Gray zone:**
- Disagreement or debate → extract as PROBLEM or DISPUTED, note both positions
- "I'm not sure..." → extract with `confidence: <70`
- Tangential but interesting → extract if it clarifies an existing item
