## THE EVOLUTION PATTERN

### Phase 1: The Monolith (Days 1-7)
**Start simple. Almost embarrassingly simple.**

```
project/
├── main.py              # Everything in one file
├── requirements.txt
└── README.md
```

**Why this works:**
- You don't know what you're building yet
- Premature organization is technical debt
- Easy to refactor one file
- Forces you to understand the problem first

**When to evolve:** When `main.py` hits ~500-1000 lines OR you notice logical groupings forming.

---

### Phase 2: Functional Separation (Weeks 1-3)
**Split by what things DO, not what they ARE.**

```
project/
├── main.py              # Entry point only
├── core.py              # Core logic
├── database.py          # Data persistence
├── api.py               # External interfaces
├── utils.py             # Helpers
└── config.py            # Configuration
```

**Why this works:**
- Natural boundaries emerged from usage
- Each file has clear responsibility
- Still flat - easy to navigate
- Refactoring is straightforward

**When to evolve:** When files hit 500+ lines OR you see categories forming (multiple API files, multiple database handlers, etc.).

---

### Phase 3: Domain Boundaries (Months 1-3)
**Group by problem domain when complexity demands it.**

```
project/
├── src/
│   ├── authentication/
│   │   ├── __init__.py
│   │   ├── auth.py
│   │   └── tokens.py
│   ├── users/
│   │   ├── __init__.py
│   │   ├── models.py
│   │   └── service.py
│   ├── api/
│   │   ├── __init__.py
│   │   └── routes.py
│   └── database/
│       ├── __init__.py
│       └── connection.py
├── tests/
├── docs/
├── main.py
└── config.py
```

**Why this works:**
- Each domain is independently understandable
- Teams can own domains
- Testing boundaries are clear
- Onboarding is easier (explore one domain at a time)

**When to evolve:** When you have multiple engineers, or when domains grow large enough to need sub-organization.

---

### Phase 4: Architectural Separation (Months 3-12)
**Separate by architectural layer when scaling demands it.**

```
project/
├── domain/              # Business logic (pure)
│   ├── models/
│   ├── services/
│   └── interfaces/
├── infrastructure/      # External concerns
│   ├── database/
│   ├── cache/
│   └── messaging/
├── application/         # Orchestration
│   ├── commands/
│   └── queries/
├── presentation/        # User interfaces
│   ├── api/
│   ├── cli/
│   └── web/
├── shared/              # Cross-cutting
│   ├── types/
│   └── utils/
└── tests/
```

**Why this works:**
- Clear dependency flow (domain → application → infrastructure)
- Core logic isolated from external changes
- Easy to swap infrastructure (different DB, different API framework)
- Testing pure business logic is trivial

**When to evolve:** When you need multiple teams, or preparing for major technology changes.

---

## THE FORCES THAT DRIVE STRUCTURE

### 1. **Team Size**
- 1 person → Flat is fine
- 2-5 people → Domain boundaries help
- 5+ people → Architectural layers prevent collisions

### 2. **Code Volume**
- <1000 lines → One file is honest
- 1000-10,000 lines → Functional separation
- 10,000-50,000 lines → Domain boundaries
- 50,000+ lines → Architectural layers

### 3. **Change Frequency**
**High-change areas → Isolate them**

If you're constantly tweaking the API but rarely touching the database, separate them clearly so API changes don't ripple.

### 4. **Testing Needs**
**Hard to test → Pull it out**

If you can't test something without spinning up the whole system, that's a signal to extract and isolate it.

### 5. **Deployment Strategy**
- Monolith → Organize by domain
- Microservices → Each service is a separate structure
- Serverless → Organize by function boundaries

## COMMON PATTERNS & WHY THEY EXIST

### The `src/` Directory
**Why:** Separates code from tooling/config  
**When:** Python projects, prevents import conflicts  
**Example:**
```
project/
├── src/           # Importable code
│   └── myapp/
├── tests/         # Test code
├── docs/          # Documentation
└── scripts/       # Tooling
```

### The `shared/` or `common/` Directory
**Why:** Code used across multiple domains  
**Danger:** Can become a junk drawer  
**Rule:** If it's used by 3+ domains, it goes here. If it's used by 2, keep it local.

### The `interfaces/` Directory
**Why:** Separate external contracts from implementation  
**Pattern:** Dependency inversion - core logic depends on interfaces, implementations depend on core  
**Example:**
```
src/
├── domain/
│   └── user_service.py      # Depends on IUserRepository (interface)
└── infrastructure/
    └── postgres_user_repo.py # Implements IUserRepository
```

### The `tests/` Structure Mirrors Code
```
src/
├── authentication/
│   └── auth.py
└── users/
    └── service.py

tests/
├── authentication/
│   └── test_auth.py
└── users/
    └── test_service.py
```
**Why:** Easy to find tests for specific code  
**Rule:** Test structure mirrors implementation structure

## REFACTORING TRIGGERS

### When to Reorganize:

**1. The "Where Does This Go?" Moment**
- If you pause for 30+ seconds deciding where to put new code → structure is wrong
- If the "obvious" place doesn't exist → create it
- If there are 3 "obvious" places → domains aren't clear

**2. The "God Module" Smell**
- One file/directory touched by every PR → too much responsibility
- Solution: Extract concerns

**3. The "Shotgun Surgery" Smell**
- One feature change requires editing 10+ files → coupling is too high
- Solution: Group related changes closer together

**4. The "Nobody Knows" Smell**
- New team member can't find code → navigation is broken
- Solution: Clearer domain boundaries, better naming

**5. The "Import Hell" Smell**
- Circular imports, or imports that seem "wrong" → dependency flow is inverted
- Solution: Introduce layers, clarify dependencies

## WHAT NOT TO DO

### ❌ **Premature Layering**
```
project/
├── domain/
├── infrastructure/
├── application/
└── presentation/
```
**On day 1** with 100 lines of code? Overkill. You don't know your domains yet.

### ❌ **Over-Nesting**
```
project/
└── src/
    └── app/
        └── core/
            └── domain/
                └── services/
                    └── user/
                        └── authentication/
                            └── login.py  # 7 levels deep!
```
**Rule of thumb:** 3-4 levels max. If you need more, your abstractions are wrong.

### ❌ **Framework Worship**
Don't organize around framework concepts unless you're building a framework.

**Bad:**
```
project/
├── models/         # Django pattern
├── views/
└── templates/
```
**Better:**
```
project/
├── users/          # Domain pattern
├── products/
└── orders/
```

### ❌ **Junk Drawers**
```
project/
├── utils/          # 50 unrelated functions
├── helpers/        # 30 random classes
└── misc/           # Nobody knows what's here
```

**Fix:** If a utility is only used by one domain, move it there. If used by many, name it specifically: `date_utils.py`, `string_formatters.py`.
