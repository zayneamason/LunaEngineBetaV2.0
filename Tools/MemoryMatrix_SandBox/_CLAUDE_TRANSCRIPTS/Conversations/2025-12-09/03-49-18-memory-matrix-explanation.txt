---

## Core Architecture

**Hybrid design**: Timeline + Graph memory. Think of it as two ways to query the same data:

1. **SQLite + FTS5** for the timeline (when did this happen? find "VAD" discussions)
2. **NetworkX graph** for relationships (what enabled this decision? what contradicts that?)

The clever bit is they work together. Search finds nodes, then graph expansion pulls in related context.

---

## The Node Types

These are the atomic units - each memory is one of these:

| Type | What It Captures |
|------|------------------|
| **FACT** | Claims, measurements, system state |
| **DECISION** | Choices made, tradeoffs considered |
| **PROBLEM** | Bugs, gaps, things to solve |
| **ASSUMPTION** | Untested beliefs we're operating on |
| **CONNECTION** | Explicit relationships ("X enables Y") |
| **ACTION** | Tasks, todos |
| **OUTCOME** | Results of experiments/actions |

---

## How Nodes Connect

Six relationship types that form the graph edges:

- **depends_on** → X requires Y
- **enables** → X makes Y possible  
- **contradicts** → X conflicts with Y (useful for catching stale assumptions)
- **clarifies** → X explains/refines Y
- **related_to** → Weak association
- **supersedes** → X replaces Y (handles evolution)

---

## Safety Philosophy

**No deletes. Ever.** Nodes are archived, not destroyed. Foreign keys prevent orphaned edges. This is critical - memories can't just disappear.

---

## How This Maps to Your Council

Your Financial Systems Controller wants audit trails? The matrix keeps everything with timestamps, confidence scores, and status tracking. Your Enterprise Risk Manager wants to trace why decisions were made? `trace_dependencies()` walks the graph backwards through `depends_on` edges.

## OFX vs CSV

| Aspect | CSV | OFX/QFX (XML) |
|--------|-----|---------------|
| Structure | Flat, varies by bank | Standardized schema |
| Account metadata | None | Included (account #, type, bank ID) |
| Transaction IDs | None | `FITID` - unique per transaction |
| Deduplication | Have to hash it yourself | Built-in unique IDs |
| Date handling | String parsing | ISO format, consistent |
| Balance info | None | Opening/closing balances included |

**OFX wins.** The `FITID` field alone solves the deduplication problem - you can re-import without worrying about duplicates.

"""Import Apple Cash Excel statement into financial_matrix."""

import sys

from datetime import datetime
import pandas as pd
import re

def import_apple_cash(db, xlsx_path: str, account_number: str = "5364", 
                      account_name: str = "Apple Cash", account_type: str = "wallet"):
    df = pd.read_excel(xlsx_path)
    db.conn.execute("INSERT OR IGNORE INTO accounts (account_number, name, type) VALUES (?, ?, ?)",
                    (account_number, account_name, account_type))
    imported = skipped = 0
    for i in range(32, len(df)):
        row = df.iloc[i]
        date_val, description, amount, balance = row.iloc[0], row.iloc[1], row.iloc[7], row.iloc[9]
        if not isinstance(date_val, datetime) or pd.isna(amount):
            continue
        date_str = date_val.strftime("%Y-%m-%d")
        description = str(description).replace('\n', ' ').strip() if pd.notna(description) else ""
        if db.conn.execute("SELECT 1 FROM transactions WHERE account_number=? AND date=? AND amount=? AND description=?",
                          (account_number, date_str, float(amount), description)).fetchone():
            skipped += 1
            continue
        party = None
        if "Payment to" in description or "Received from" in description:
            match = re.search(r'\+\d{11}', description)
            if match: party = match.group()
        category = None
        if amount > 0: category = "income:apple_cash"

                       (account_number, date_str, description, float(amount), float(balance) if pd.notna(balance) else None, category, party))
        imported += 1
    db.conn.commit()
    return imported, skipped

if __name__ == "__main__":

import re
