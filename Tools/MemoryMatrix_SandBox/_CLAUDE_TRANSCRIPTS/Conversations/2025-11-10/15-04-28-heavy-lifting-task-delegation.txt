## **3. CONVERSATION STATE MANAGEMENT**

### **Why You Need State**

**Without state:**
```
User: "Create a deck project for Martinez"
AI: "What's the budget?"
User: "$18,500"
AI: "I don't understand. Please provide project details."  ← BROKEN
```

**With state:**
```
User: "Create a deck project for Martinez"
AI: "What's the budget?"  [Stores: awaiting=budget, partial_data={client:"Martinez", type:"Deck"}]
User: "$18,500"  [Retrieves state, adds budget, completes creation]
AI: "✅ Created Martinez Deck project with $18,500 budget"
```

### **State Data Structure**

Store in Make.com **Data Store** module:

```json
{
  "conversation_id": "telegram_user_id",
  "timestamp": "2024-11-10T10:30:00Z",
  "state": "AWAITING_RESPONSE",  // IDLE|AWAITING_RESPONSE|CONFIRMING|PROCESSING
  "intent": "CREATE",  // CREATE|UPDATE|QUERY|null
  "awaiting_field": "Budget",  // Which field we asked about
  "partial_data": {
    // Accumulated data so far
    "Project Name": "Martinez Deck",
    "Client Name": "Martinez",
    "Project Type": "Deck",
    "Priority": "High"
    // Budget still missing
  },
  "last_action": {
    "type": "CREATE_INITIATED",
    "notion_page_id": null,  // Set after creation
    "notion_page_url": null
  },
  "clarification_attempts": 0  // Track how many times we've asked
}
```

### **State Lifecycle**

```mermaid
stateDiagram-v2
    [*] --> IDLE: New conversation
    IDLE --> PROCESSING: Message received
    PROCESSING --> AWAITING_RESPONSE: Missing data
    PROCESSING --> CONFIRMING: Action ready
    AWAITING_RESPONSE --> PROCESSING: User responds
    CONFIRMING --> IDLE: User confirms
    CONFIRMING --> PROCESSING: User requests edits
    CONFIRMING --> IDLE: User cancels
    IDLE --> [*]: Timeout (1 hour)
```

### **State Management Rules**

1. **On new message:** Check if conversation_id exists in Data Store
2. **If state = AWAITING_RESPONSE:** Treat message as answer to last question
3. **If state = CONFIRMING:** Treat message as confirmation/edit/cancel
4. **If state = IDLE or not found:** Classify as new intent
5. **After successful action:** Clear state to IDLE
6. **After 1 hour of inactivity:** Auto-clear state

## **8. ERROR RECOVERY SYSTEM**

### **Error Types & Handlers**

#### **1. Notion API Errors**

**Common Errors:**
- Rate limit exceeded (429)
- Invalid database ID (404)
- Invalid property value (400)
- Network timeout

**Recovery Strategy:**

**Module Template: Error Handler**
```
Type: Error Handler (attached to Notion modules)

If error.statusCode = 429:
  - Wait 60 seconds
  - Retry (max 3 attempts)
  - If still fails, return to Master Agent with error

If error.statusCode = 404:
  - Log error
  - Return to Master Agent: "Database configuration error"
  
If error.statusCode = 400:
  - Parse error message
  - Return to Master Agent with specific field error
  
If timeout:
  - Retry once
  - If fails, return to Master Agent: "Notion is slow, try again in a moment"
```

---

#### **2. Claude API Errors**

**Common Errors:**
- Malformed JSON response
- Context length exceeded
- API key invalid
- Rate limit

**Recovery Strategy:**

**Module: JSON Parse Error Handler**
```
Type: Error Handler (attached to JSON Parse modules)

Action: Claude API Call
Prompt: "The previous response wasn't valid JSON. Extract the data and return ONLY valid JSON with no markdown code blocks."

If still fails:
  - Return to Master Agent: "I'm having trouble processing that. Can you rephrase?"
```

---

#### **3. Validation Errors**

**Common Errors:**
- Invalid Status value
- Invalid Project Type
- Missing required fields
- Invalid date format

**Recovery Strategy:**

**Built into validation prompts:**
```
- Apply defaults for invalid values
- Flag errors in validation_errors array
- Return errors to Master Agent
- Master Agent asks for clarification
```

---

#### **4. State Management Errors**

**Common Errors:**
- Data Store not found
- Stale state (>1 hour old)
- Corrupted state data

**Recovery Strategy:**

**Module: State Check with Error Handling**
```
If state not found:
  - Treat as new conversation
  
If state.timestamp > 1 hour ago:
  - Clear state
  - Treat as new conversation
  - Notify user: "Session expired, starting fresh"

If state data malformed:
  - Clear state
  - Log error
  - Treat as new conversation
```

### **Step 8: Monitor & Debug**

**Make.com Monitoring:**
1. Check scenario execution history
2. Look for errors (red flags)
3. Click executions to see module outputs
4. Use "Download execution bundles" for debugging

**Common Issues:**

❌ **"Webhook response timeout"**
- Handler took >30 seconds
- Increase timeout in webhook settings
- Or: Return immediately, process async

❌ **"Invalid Notion database ID"**
- Database not shared with integration
- Or: Wrong ID format (use full UUID)

❌ **"Claude returned invalid JSON"**
- Check prompt for "no markdown" instruction
- Add JSON parser error handler
- Re-prompt Claude to fix output

❌ **"State not persisting"**
- Data Store not configured
- Or: Wrong key (conversation_id)
- Check Data Store records manually

### **Step 10: Scale Considerations**

**When you hit limits:**

**Make.com:**
- Free plan: 1,000 operations/month
- Core plan: 10,000 ops/month ($9)
- Pro plan: 40,000 ops/month ($29)
- Each module execution = 1 operation
- Average flow: ~15-20 operations per message

**Claude API:**
- Sonnet 3.5: $3 per million input tokens, $15 per million output
- Average message: ~2000 input tokens, ~500 output tokens
- Cost per message: ~$0.01-0.02
- Budget: $100/month = ~5,000-10,000 messages

**Notion API:**
- Rate limit: 3 requests/second
- Use Make.com sleep modules if hitting limit
- Batch operations when possible

**Scaling Strategy:**
1. Start with 1 contractor (Mike)
2. Monitor costs per contractor
3. Scale to 5 contractors (phase 1)
4. Optimize prompts to reduce tokens
5. Consider moving to self-hosted (Supabase + Railway) if costs exceed $200/month
