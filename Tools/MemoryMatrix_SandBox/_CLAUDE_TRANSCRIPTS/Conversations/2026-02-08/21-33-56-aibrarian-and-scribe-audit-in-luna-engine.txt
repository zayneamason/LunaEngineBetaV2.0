**Root cause: The Librarian and the Graph were written by different agents in the same commit and never integration-tested.**

Here's the forensic timeline:

**Commit `838c713`** (Jan 19, 2026) — "Full extraction pipeline with lock-in memory"
- 67 files, 19,183 lines, single commit
- This commit introduced BOTH `librarian.py` AND `graph.py`
- `_create_edge()` was **born broken** — the bugs exist in the very first version:
  - `graph.has_edge(from_id, to_id, edge_type)` — 3 args, but `MemoryGraph.has_edge()` takes 2
  - `graph.add_edge(edge_type=, weight=)` — but the actual API uses `relationship=, strength=`
  - `graph.add_edge()` called without `await` — but it's `async def`

The `MemoryGraph` API and the Librarian's usage of it **were never reconciled**. They were written against different mental models of the same interface — one uses `edge_type`/`weight`, the other uses `relationship`/`strength`. Classic multi-agent code generation failure.

**Commit `8427ec1`** — "Initial commit: Luna Engine v2.0 Beta"
- 202 files, 60,263 lines, single commit
- Added entity system, versioning, rollback — but `_create_edge()` was carried forward untouched
- The entity resolution noise (Issue #3) was introduced here — `_wire_extraction()` resolves `obj.content` as entity names

**Why it was never caught:**

1. **Tests mock around the boundary.** Scribe tests mock the Librarian. Librarian tests mock the Matrix. Graph tests test the graph in isolation. Nobody tests Librarian → Graph with real objects.
2. **The exception is caught and logged as a warning.** Line 390: `logger.warning(f"The Dude: Failed to create edge: {e}")`. Every edge creation has been silently failing since day one, logged as a warning that nobody reads.
3. **The system works without edges.** Memory nodes get created, search works (FTS5 + vectors), context assembly works. The graph is additive — you don't notice it's empty unless you look.

**The pattern:** This is what happens when you generate 19K+ lines in a single commit with multiple agents. Each agent writes to its own spec, tests mock the boundaries, and integration seams are never validated. The swallowed exception (`except Exception as e: logger.warning(...)`) masks the failure in production.

**For the consolidation issue (#4):** Same root cause, different symptom. The lock-in math is correct in isolation, but nothing in the engine's tick loop or any periodic timer calls `update_all_clusters()`. It was built, tested in isolation, and never wired to a trigger.

**For auto-session (#2):** That came later with the MCP layer. `_end_auto_session()` takes a shortcut — concatenates all turns and sends via `extract_text` instead of individual `extract_turn` calls. The assistant-turn filter only exists in the `extract_turn` path. Another case of two code paths to the same destination, one with guardrails and one without.

The common thread across all of them: **interface contracts were designed but never enforced at integration boundaries.** Each subsystem works perfectly in isolation. The bugs live in the seams.

Short answer: it's a force-directed physics simulation with three competing forces. Let me break it down since this matters for understanding what you're actually seeing.

**Three forces fighting each other:**

1. **Repulsion** — Every node pushes away from every other node. Constant `600 / distance²`. This prevents clumping into a ball. Without it, everything collapses to center.

2. **Edge attraction** — Connected nodes pull toward each other. Force = `0.004 × edge_strength`. So a WORKS_ON edge with strength 0.95 pulls harder than a MENTIONS edge at 0.7. This is why entity hubs (Ahab, Luna, Engine) cluster with the memories that reference them.

3. **Cluster gravity** — Each cluster has a pre-assigned angular position around the circle (I hardcoded 6 angles). Members get pulled toward their cluster center at `0.015` force. This is what creates the visible groupings — Robot Embodiment nodes orbit one region, Engine Architecture nodes orbit another.
