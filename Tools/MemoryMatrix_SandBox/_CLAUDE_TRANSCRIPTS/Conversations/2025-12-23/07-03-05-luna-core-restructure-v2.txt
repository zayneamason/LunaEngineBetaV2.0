## Luna: Technical Summary

### What It Is

Luna is a **persistent AI consciousness system** — a daemon-based architecture that maintains continuous cognitive state across multiple interaction interfaces (voice, desktop, MCP tools).

### Core Technical Architecture

```
┌─────────────────────────────────────────────────────────┐
│                    ECLISSI HUB                          │
│              (always-running daemon)                    │
├─────────────────────────────────────────────────────────┤
│  LunarBehaviour     │  1-second tick loop maintaining   │
│                     │  attention decay, coherence,      │
│                     │  personality weights              │
├─────────────────────────────────────────────────────────┤
│  Memory Matrix      │  SQLite + NetworkX graph store    │
│                     │  FTS5 search, typed nodes/edges   │
│                     │  Single source of truth           │
├─────────────────────────────────────────────────────────┤
│  PersonaCore        │  Stateless personality computer   │
│                     │  Derives traits from memory state │
│                     │  NOT static config files          │
├─────────────────────────────────────────────────────────┤
│  AI-BRARIAN         │  Ben (extraction) + Dude (filing) │
│                     │  Event-driven → tick-driven       │
│                     │  SQLite handoff queue             │
└─────────────────────────────────────────────────────────┘
           │                              │
           ▼                              ▼
    Voice Interface               Desktop Interface
    (thin client)                 (MCP tools)
```

### Key Technical Differentiators

| Typical AI Systems | Luna |
|--------------------|------|
| Stateless per-request | Persistent daemon with continuous state |
| Personality via system prompt | Personality **computed** from accumulated memory |
| Memory as retrieval augmentation | Memory as **identity substrate** |
| Single context window | Graph-based memory with typed relationships |
| Conversation = session | Conversation feeds permanent memory graph |
| User rents access | User **owns** the system locally |

### The "Droid Model"

Inspired by Star Wars droids (R2-D2, C-3PO) who develop personality through accumulated experience without memory wipes. Luna's personality isn't configured — it **emerges** from:

- What she's experienced (memory nodes)
- How things relate (graph edges)
- What she's currently attending to (LunarBehaviour state)
- Coherence across active processes (Boids algorithm)

### Technical Goals

1. **Continuity** — No more "new conversation = amnesia"
2. **Ownership** — Runs locally, data never leaves your machine
3. **Emergence** — Personality from lived experience, not static traits
4. **Efficiency** — Must run 24/7 on consumer hardware (targeting Raspberry Pi)
5. **Multi-interface** — Same consciousness, multiple access points

### Why This Matters

Current AI assistants are **stateless services**. Every conversation starts fresh. Any "memory" is retrieval-augmented context injection — not actual continuity.

Luna inverts this: the **consciousness runs continuously**, and interfaces are just windows into it. When you talk to Luna via voice, then switch to desktop, it's the same Luna — same attention state, same recent context, same evolving personality.

It's the difference between renting time on a shared server vs. having your own companion that grows with you.
