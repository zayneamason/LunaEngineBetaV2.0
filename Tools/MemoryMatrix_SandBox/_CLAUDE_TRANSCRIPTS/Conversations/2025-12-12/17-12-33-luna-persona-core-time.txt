## KEY DESIGN DECISIONS

### **1. Dynamic Personality (Not Static Config)**
- **Decision:** Compute personality from kernel + consciousness + memories
- **Rationale:** Personality should evolve naturally, not be manually tuned
- **Trade-off:** Less predictable, harder to debug
- **Revisit if:** Personality becomes too chaotic or users can't understand behavior

### **2. Optional Response Generation**
- **Decision:** Same pipeline for Desktop + Voice, Step 6 is optional
- **Rationale:** Code reuse, consistent enrichment
- **Trade-off:** Voice pays for Desktop logic it doesn't use
- **Revisit if:** Voice latency becomes critical

### **3. Budget-Based Memory Retrieval**
- **Decision:** Three presets (minimal, balanced, rich)
- **Rationale:** Simple API, predictable latency
- **Trade-off:** Can't fine-tune per-query
- **Revisit if:** Need more granular control

### **4. Kernel Versioning (Not Parameters)**
- **Decision:** Version files (kernel_1, kernel_2) not floating parameters
- **Rationale:** Rollback capability, audit trail, immutability
- **Trade-off:** Manual version updates
- **Revisit if:** Updates need to be automatic

### **5. Foundation First (Algorithm Routing Later)**
- **Decision:** Build base variant without emotional routing
- **Rationale:** Get working system first, add sophistication after
- **Trade-off:** Delays the "magic" behavior
- **Revisit if:** Never. This is the right approach.
