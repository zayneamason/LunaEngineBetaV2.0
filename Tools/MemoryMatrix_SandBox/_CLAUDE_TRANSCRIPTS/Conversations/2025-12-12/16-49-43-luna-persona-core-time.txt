## What Smells Wrong

### 1. **Memory Pattern Analysis is Too Crude**
```python
# Scans last 50 memory nodes for keyword signals
# Each match adds +0.01 to trait (capped at ±0.2)
```

**Problem:** Keyword matching with linear accumulation is brittle.
- "code" appears in 20 memories → +0.2 to technical_depth regardless of context
- Doesn't distinguish between "I love debugging code" and "this code is broken again"
- Cap at ±0.2 means diminishing returns after 20 matches

**Why it'll hurt:** Personality will feel laggy and unresponsive to recent shifts in conversation tone. You'd need 20+ memories mentioning "fun" to max out playfulness.

**Fix direction:** Weight by recency (exponential decay), consider sentiment/context, or use embedding similarity instead of keyword grep.

---

### 2. **P1 Safety Check is Hacky**
```python
if self._is_identity_query(query) and not self._has_meaningful_memory(context):
    return random.choice(MEMORY_COMPROMISED_RESPONSES)
    # e.g., "*PFFFRRRRTTTT* ...that wasn't me"
```

**Problem:** This is clever but fragile.
- What qualifies as "identity query"? Pattern matching on "who are you"? What about "tell me about yourself" or "what's your deal"?
- What's "meaningful memory"? Count > 0? Or semantic relevance?
- Fart noise fallback is funny but breaks immersion if it triggers incorrectly.

**Why it'll hurt:** Edge cases. "What do you remember about our project?" could trigger this if "project" doesn't match memory keywords, even though memories exist.

**Fix direction:** Make the identity check more specific (whitelist phrases) and memory check more semantic (embeddings, not just count).

---

### 3. **Timing Assumptions Won't Scale**
```
Memory Matrix smart_fetch: ~50-150ms
```

**Problem:** This assumes small graph (hundreds of nodes). What happens at 10k? 100k?
- FTS5 search scales well, but graph expansion is O(nodes × avg_degree)
- "For each found node, traverse relationships" could explode with dense graphs

**Why it'll hurt:** As Luna accumulates memories over months, query latency will creep up. Voice's 500-800ms budget will become 1-2 seconds.

**Fix direction:** Limit graph expansion depth explicitly (2 hops max?), add early termination, or precompute clusters.

---

### 4. **Tight Coupling = Testing Pain**
PersonaCore depends on:
- Memory Matrix
- LunarBehaviour
- StateMachine  
- PersonalityComputer
- ResponseGenerator

**Problem:** Can't test PersonaCore without mocking 5 dependencies. Any change to Memory Matrix interface breaks PersonaCore tests.

**Why it'll hurt:** Refactoring becomes expensive. Want to swap FTS5 for embeddings? Now you're touching PersonaCore integration code.

**Fix direction:** Define interfaces/protocols. PersonaCore should depend on `IMemoryProvider`, not `MemoryMatrix` directly. Dependency injection at construction.

---

### 5. **No Error Recovery Strategy**
What happens if:
- Memory Matrix is down?
- Claude API times out?
- Consciousness state fetch fails?

**Problem:** The doc doesn't say. Does PersonaCore fail hard? Return degraded context? Retry?

**Why it'll hurt:** In production, dependencies fail. If Memory Matrix goes down, does Voice Luna stop working entirely, or does it respond with "I can't access my memories right now"?

**Fix direction:** Graceful degradation. Each dependency should have a fallback mode. No memories? Say so. No API? Return canned response.

## Structural Questions

### Budget System is Inflexible
```python
BUDGET_PRESETS = {
    "minimal": 1800,
    "balanced": 3800,  
    "rich": 7200
}
```

**Question:** What if a query needs 2500 tokens? Or what if certain query types (e.g., "tell me everything about X") should auto-escalate budget?

**Why it matters:** Three fixed presets force users to pick "close enough." Can't optimize per-query.

**Possible fix:** Add `budget: int | str` param. Strings = presets, ints = exact token count. Or add heuristics to auto-escalate based on query structure.

---

### Response Generation is Doing Too Much (Step 7)
```
- P1 safety checks
- Build system prompt  
- Call Claude API
- Record debug telemetry
- Return response + debug info
```

**Question:** That's 4 responsibilities in one component. What if you want to swap safety checks? Or change prompt templating?

**Why it matters:** Single Responsibility Principle. Each concern should be independently testable and modifiable.

**Possible fix:** Break into stages:
- `SafetyValidator.check(query, context)`
- `PromptBuilder.build(context)`
- `APIClient.call(prompt)`
- `TelemetryRecorder.log(response)`

---

### Memory Feedback Loop Stability
**Question:** Responses generate memories (presumably via extraction), which affect future personality computation. Is this loop stable or could it drift?

Example: Luna responds with lots of jokes → jokes get stored as memories → PersonalityComputer sees "joke" keywords → boosts humor trait → Luna responds with more jokes → ...

**Why it matters:** Positive feedback loops can runaway. Personality could drift toward extremes over weeks.

**Missing:** Any damping/smoothing mechanism mentioned in the doc.
