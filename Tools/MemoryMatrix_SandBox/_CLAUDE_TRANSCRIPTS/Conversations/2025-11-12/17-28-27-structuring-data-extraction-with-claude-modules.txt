## Core Architecture Concept

Instead of one giant router trying to do everything, you want:

**Pipeline: Message → Categorization → Extraction → Validation → Notion**

Each step is a separate Claude API call with a focused job.

## Proposed Module Breakdown

### **Module 1: Intent Classifier**
**Job:** Figure out what the contractor wants
**Input:** Raw message text
**Output:** Single intent category + confidence score

```json
{
  "intent": "CREATE_PROJECT",
  "confidence": 0.95,
  "reasoning": "Message contains client name, project type, and budget"
}
```

**Possible Intents:**
- `CREATE_PROJECT`
- `UPDATE_PROJECT` 
- `QUERY_INFO`
- `ADD_TASK`
- `UPDATE_TASK`
- `LOG_NOTE`
- `UNCLEAR` (triggers follow-up)

### **Module 2: Entity Extractor**
**Job:** Pull out all the facts, regardless of intent
**Input:** Raw message + detected intent
**Output:** Structured key-value pairs

```json
{
  "client_name": "Sarah Johnson",
  "project_type": "Kitchen Remodel",
  "budget": 45000,
  "timeline": "8 weeks",
  "location": "123 Oak Street",
  "scope_items": ["cabinets", "countertops", "backsplash"],
  "missing_fields": ["start_date", "permit_requirements"]
}
```

**Key Feature:** Returns `null` for missing data, doesn't hallucinate

### **Module 3: Field Mapper**
**Job:** Match extracted entities to exact Notion database fields
**Input:** Extracted entities + Notion schema
**Output:** Notion-ready payload

```json
{
  "database_id": "8619343bbb324c9d87c1d0300d4df837",
  "properties": {
    "Project Name": {
      "title": [{"text": {"content": "Johnson Kitchen Remodel"}}]
    },
    "Client": {
      "relation": [{"id": "client-uuid-from-search"}]
    },
    "Budget": {
      "number": 45000
    },
    "Timeline": {
      "rich_text": [{"text": {"content": "8 weeks"}}]
    }
  }
}
```

### **Module 4: Validator**
**Job:** Check if we have minimum required fields
**Input:** Mapped fields + business rules
**Output:** Ready to send OR list of missing critical info

```json
{
  "ready": false,
  "missing_critical": ["start_date"],
  "missing_optional": ["permit_requirements", "special_requests"],
  "suggested_followup": "When do you want to start the Johnson project?"
}
```

## Make.com Workflow Structure

```
1. [Webhook] Receive WhatsApp message
2. [Claude API] Module 1: Intent Classifier
3. [Router] Branch based on intent
4. [Claude API] Module 2: Entity Extractor
5. [Claude API] Module 3: Field Mapper
6. [Claude API] Module 4: Validator
7. [Branch] 
   - If ready → Create Notion page
   - If missing info → Send follow-up question
```

## Why This Works Better

**Single Responsibility:**
- Each module has ONE job
- Easier to debug when something breaks
- Can test modules independently

**Clean Variable Flow:**
- Module 1 output becomes Module 2 input
- No complex nested parsing
- Make.com can easily map between steps

**Maintainable:**
- Update one module without touching others
- Add new intents without rewriting everything
- Swap out modules (e.g., replace Entity Extractor with better version)

**Debuggable:**
- See exactly which module produced bad output
- Test with real contractor messages module by module
- Iterate on prompts without breaking the whole system

   - MERGED_PARTIAL_DATA: Combined with existing data
   - UPDATED_AWAITING_FIELD: Answered a specific question

## **Why This Works**

Make.com scenarios are **linear** - each module waits for the previous one to finish.

So by the time you reach the JSON Aggregator:
- ✅ Module 1 has already run and stored its outputs
- ✅ Module 2 has already run and stored its outputs
- ✅ Sheets search has already run and stored its outputs

The aggregator just **references** their stored outputs - it doesn't re-run them.
