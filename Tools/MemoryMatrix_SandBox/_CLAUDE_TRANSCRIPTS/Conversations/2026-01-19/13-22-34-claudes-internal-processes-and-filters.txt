Ahab is asking what happens if we remove FTS5 from the equation. Let me think about what FTS5 actually provides and what alternatives exist.

FTS5 is SQLite's full-text search extension. It provides:
1. Fast keyword/phrase search with tokenization
2. Boolean operators (AND, OR, NOT)
3. Prefix matching
4. Ranking by BM25

Without FTS5, the alternatives are:
1. LIKE queries - what Luna Engine currently uses. Slow (O(n) scan), no ranking, no tokenization
2. Pure semantic search (sqlite-vec embeddings) - good for conceptual similarity but misses exact matches
3. External search like Elasticsearch/Meilisearch - overkill, adds dependency

The hybrid search in Eclissi uses:
- FTS5 for keyword matches (BM25 ranking)
- sqlite-vec for semantic similarity
- RRF (Reciprocal Rank Fusion) to combine both

If we remove FTS5:
- Lose fast keyword search
- Lose BM25 ranking
- Can still do semantic search with embeddings
- Can still do exact match with LIKE (slow)

BUT - here's the interesting question: do we actually need keyword search if we have good semantic search?

Arguments for keeping FTS5:
- Exact term matching ("what did I say about FAISS")
- Fast - O(1) index lookup vs O(n) scan
- Complements semantic search (sometimes you want literal matches)

Arguments against FTS5:
- Complexity
- Another thing to maintain
- Semantic search might be good enough for most queries
- Luna Engine is trying to be simpler

The real question: what's the actual retrieval pattern?
- If queries are conceptual ("what do I know about memory systems") → semantic wins
- If queries are literal ("find mentions of sqlite-vec") → keyword wins
- If queries are mixed → hybrid wins

For Luna's use case, most queries are probably conceptual. The Scribe extracts structured facts, the Librarian files them. 

Most retrieval is probably asking "what's relevant to this concept" rather than hunting for exact phrases. The main risk is missing specific terminology—like a query about FAISS might not surface a node that uses that exact term if the embedding doesn't capture it well. I could mitigate this by keeping LIKE as a fallback for literal term matching when semantic search alone feels insufficient.Good question. Let's break down what FTS5 actually gives us:

## What FTS5 Provides

| Capability | FTS5 | Without FTS5 |
|------------|------|--------------|
| Keyword search | O(1) index lookup | O(n) table scan (LIKE) |
| Phrase matching | `"exact phrase"` | Regex or multiple LIKEs |
| Boolean ops | `term1 AND term2 NOT term3` | Multiple queries |
| Prefix search | `memor*` | `LIKE 'memor%'` |
| BM25 ranking | Built-in relevance scoring | None |
| Tokenization | Handles word boundaries | Manual |

## The Hybrid Search Flow (Current Eclissi)

```
Query: "What did I decide about vector search?"
        │
        ├── FTS5 path → finds "decided to use FAISS"
        │                (keyword match on "decided", "vector", "search")
        │
        ├── Semantic path → finds conceptually similar nodes
        │                   (embeddings capture meaning)
        │
        └── RRF merge → combines both, dedupes, ranks
```

## Without FTS5

```
Query: "What did I decide about vector search?"
        │
        ├── Semantic ONLY → finds conceptually similar nodes
        │
        └── LIKE fallback → slow, no ranking, exact substring only
```

## What You Lose

1. **Speed** - FTS5 is O(1), LIKE is O(n). With 14k nodes, LIKE scans everything.
2. **Exact term recall** - "Find mentions of FAISS" might miss if embedding doesn't capture the acronym well
3. **Hybrid precision** - RRF combining keyword + semantic is more accurate than either alone

## What You Keep

1. **Semantic search** - sqlite-vec still works, conceptual matching
2. **Simplicity** - One search path instead of two
3. **Less schema complexity** - No FTS5 virtual tables to maintain

## My Take

For Luna's use case, **semantic-only might be fine**:

- Scribe extracts structured facts → content is clean
- Queries are usually conceptual ("what's relevant to X") not literal ("find exact phrase Y")
- 14k nodes isn't huge — even LIKE is tolerable at that scale

**The risk**: Exact term queries suffer. "What did I say about sqlite-vec?" might miss nodes if the embedding doesn't capture the exact term.

**Mitigation**: Keep LIKE as an explicit fallback for literal queries. Route at query time:
```python
if query.startswith('"') or query.has_exact_flag:
    use_like_search()
else:
    use_semantic_search()
```

**Bottom line**: Removing FTS5 simplifies the stack. You trade keyword precision for simplicity. If most queries are conceptual, it's probably fine. If you find yourself missing exact matches, add it back later.

```python
async def keyword_search(self, query: str, limit: int = 20) -> list[str]:
    """FTS5 keyword search, returns node IDs ranked by BM25."""
    rows = await self.db.fetchall("""
        SELECT rowid, bm25(memory_nodes_fts) as score
        FROM memory_nodes_fts
        WHERE memory_nodes_fts MATCH ?
        ORDER BY score
        LIMIT ?
    """, (query, limit))
    return [row[0] for row in rows]
```

```python
async def hybrid_search(self, query: str, limit: int = 20) -> list[str]:
    keyword_results = await self.keyword_search(query, limit=50)

    return reciprocal_rank_fusion(keyword_results, semantic_results, k=60)[:limit]
```

## Lock-In Coefficient: Luna's Memory Persistence System

### The Core Concept

The **lock-in coefficient (L)** is a value between 0.15 and 0.85 that represents how "permanent" a memory is in Luna's mind. Think of it as the neurological strength of a synapse — frequently used connections get stronger, neglected ones fade.

```
L = 0.15 ──────────────────────────────────────────── 0.85
          │                │                      │
       DRIFTING         FLUID                 SETTLED
       (< 0.30)      (0.30-0.70)              (≥ 0.70)
```

### The Three States

| State | Range | Meaning |
|-------|-------|---------|
| **DRIFTING** 🟠 | L < 0.30 | Rarely accessed, may fade over time |
| **FLUID** 🔵 | 0.30 ≤ L < 0.70 | Active but not yet core knowledge |
| **SETTLED** 🟢 | L ≥ 0.70 | Core knowledge, persistent, won't decay |

### What Affects Lock-In?

Four weighted factors feed into a sigmoid function:

```python
activity = (
    retrieval_count × 0.4 +      # How often Luna recalls this
    reinforcement_count × 0.3 +   # User explicitly marked important
    locked_neighbor_count × 0.2 + # Connected to settled memories
    locked_tag_sibling_count × 0.1  # Shares tags with settled memories
) / 10
```

Then passed through sigmoid to get L:
```python
L = 0.15 + 0.70 × sigmoid(activity)
```

### Why This Matters for Luna

**Memory Quality Control:**
- New memories start FLUID (~0.42)
- Frequently accessed memories become SETTLED
- Ignored memories drift toward 0.15 and may be pruned

**Graph Contagion:**
- If a memory's *neighbors* are settled, it gets a boost
- This creates "neighborhoods" of important knowledge
- Related concepts reinforce each other

**Pruning Decisions:**
- DRIFTING memories are candidates for synaptic pruning
- SETTLED memories are protected from cleanup
- This prevents the Memory Matrix from growing unboundedly

### Option B: First-Class Entity Table (Structured)
```sql
CREATE TABLE entities (
    id TEXT PRIMARY KEY,
    entity_type TEXT,  -- 'person', 'project', 'place'
    name TEXT,
    aliases TEXT,      -- JSON array
    metadata TEXT,     -- JSON blob
    created_at TEXT,
    updated_at TEXT
);

CREATE TABLE entity_relationships (
    from_entity TEXT,
    to_entity TEXT,
    relationship TEXT,  -- 'friend', 'collaborator', 'creator'
    strength REAL,
    notes TEXT
);
```

Then Luna can query:
```sql
SELECT * FROM entities WHERE name LIKE '%Marzipan%';
SELECT * FROM entity_relationships WHERE from_entity = 'ahab';
```
