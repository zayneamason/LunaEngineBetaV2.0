## Assessment: Context State Machine Study

### What This Is
A comprehensive design document for **context-aware personality modulation** via hot-swappable LoRA adapters. It's essentially "Luna adapts her *being*—not just her *output*—to what you're doing."

### What's Working

**1. Clean Separation of Concerns**
The architecture correctly identifies four coordinated changes per transition:
- LoRA swap (personality)
- Memory scope (what she remembers)
- Tool availability (what she can do)
- Delegation strategy (how she handles tasks)

This is the right decomposition.

**2. Detection Algorithm is Solid**
The multi-signal weighting approach (keywords 40%, temporal 20%, history 20%, projects 10%, explicit override 100%) is pragmatic. The explicit command escape hatch (`work mode`, `focus mode`) is critical for user control.

**3. The Delegation Strategies are Genuinely Different**
- **Efficiency-focused**: Execute immediately, minimal back-and-forth
- **Wellbeing-focused**: Check user state, offer support vs solutions
- **Exploration-focused**: Generate options, invite collaboration
- **Focus-preservation**: Batch interruptions, auto-execute safe ops

These aren't cosmetic—they're different *behaviors*.

### Concerns & Questions

**1. LoRA Hot-Swap Reality Check**
The doc assumes sub-second adapter swapping. Current MLX LoRA loading is... not that. Need validation:
- What's the *actual* swap time on M2/M3 with 7B model?
- Is adapter caching (keep 2-3 in memory) viable given VRAM constraints?
- Fallback to core personality if swap fails—good, but what's the user experience?

**2. Memory Scope Leakage**
The hard boundaries (work memories in work, personal in personal) are appealing but problematic:
- User asks "remember that thing from work about the deadline?" while in personal mode
- Cross-context references are *common*
- The doc mentions this in Research Questions but doesn't resolve it

**Proposal**: Instead of hard scoping, use **soft bias**—weight context-tagged memories higher, but don't hide others. User can always access anything explicitly.

**3. Training Data Chicken-and-Egg**
You need context-tagged training data to train context-specific LoRAs. But you need the system running to collect context-tagged data. The manual seed phase (50-100 hand-crafted) is the bootstrap path, but that's labor-intensive.

**4. Transition Announcement UX**
The doc shows announcements like `[Focus mode activated]`. This could be:
- Helpful orientation
- Annoying notification spam

Needs user testing. Probably should be configurable/subtle.

**5. What Happens During Transition?**
If LoRA swap takes 500ms, what does Luna do with messages that arrive during the swap? Queue? Process with old personality? This isn't specified.

### Where This Fits in Priority Stack

Given current state (local inference at 3 tok/s, personality thin, voice not implemented), this is **Phase 3 work**. The dependencies are:

1. **Persona Forge** (training data collection) → enables...
2. **Base personality LoRA** (Luna being Luna) → then...
3. **Context LoRAs** (Luna adapting to context)

You can't modulate a personality that doesn't exist yet.

### Tactical Suggestion

**Extract the State Machine + Delegation Logic NOW**, but defer LoRA swapping:

```
Phase 1 (now): State machine + memory scoping + tool filtering
              (runs on Claude API, no LoRA needed)
              
Phase 2 (after Persona Forge): Context LoRAs for local inference
```

This gets you 60% of the value (context-appropriate behavior, memory scoping, delegation strategies) without the LoRA infrastructure. The personality variation comes from system prompts tuned per context instead of LoRA weights.

---

**Bottom line**: Solid architecture, correct decomposition, but the LoRA hot-swap assumption needs validation and the full system has dependencies that aren't ready. Pull the state machine forward, defer the LoRA machinery.

What's the actual goal here? Mars College demo? If so, what specific capability does this unlock that you don't have today?
