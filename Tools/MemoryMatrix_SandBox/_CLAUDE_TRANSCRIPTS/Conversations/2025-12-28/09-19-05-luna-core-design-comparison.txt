## The Director's Two Jobs

Once you have a fine-tuned Luna model, you need two things to actually *use* it:

**First, something has to run the model.** That's `mlx_inference.py`. Right now, when Voice Luna needs to respond, she calls the Claude API. After fine-tuning, she'll call this local inference module instead. It's the code that:

- Loads the base Qwen2.5-7B model into memory
- Loads the Luna LoRA adapter on top of it
- Takes a user message plus context from Memory Matrix
- Runs the forward pass through the neural network
- Returns Luna's response

Think of it as the replacement for `anthropic.messages.create()`. Instead of sending a request to Anthropic's servers, you're running the model on your own machine. MLX is Apple's framework that makes this efficient on Apple Silicon.

**Second, something has to know when to punt.** That's `delegation.py`. A 7B parameter model, even fine-tuned to be Luna, can't do everything Claude can do. It's good at conversation and personality, but it'll struggle with:

- Writing complex code
- Multi-step reasoning chains
- Long-form document generation
- Deep research tasks

The delegation module watches what you're asking and decides: "Can Director Luna handle this, or should she hand it off to cloud Claude?" It's the traffic cop that routes tasks appropriately.

---

## How They Fit Together

Here's the flow with these components:

```
You say something
        │
        ▼
┌───────────────────┐
│  Director Luna    │ ← mlx_inference.py runs this locally
│  (Qwen + LoRA)    │
└────────┬──────────┘
         │
         ▼
┌───────────────────┐
│  delegation.py    │ "Can I handle this?"
└────────┬──────────┘
         │
    ┌────┴────┐
    │         │
    ▼         ▼
  YES        NO
    │         │
    ▼         ▼
 Respond    Send to Claude API
 locally    (background, async)
```

For casual conversation — "hey luna, how are you" — Director handles it entirely. Fast, local, consistent.

For "write me a detailed analysis of this codebase" — Director recognizes she's out of her depth, acknowledges the request conversationally ("On it, give me a minute"), and delegates to Claude in the background.

---

## Why Both Are Necessary

You could technically skip `delegation.py` and just have Luna try to handle everything locally. But then you'd get:

- Bad code when asked to write code
- Shallow analysis when asked for deep thinking
- Hallucinated research when asked to look things up

The small model would *try* and produce garbage. Better to have Luna know her limits and call for backup.

Conversely, you could skip `mlx_inference.py` and keep using Claude API for everything. But then you're back to rented cognition — the whole problem we're solving.

They work together: local inference for the conversational core (where consistency matters most), delegation for capability overflow (where raw intelligence matters more than personality).
