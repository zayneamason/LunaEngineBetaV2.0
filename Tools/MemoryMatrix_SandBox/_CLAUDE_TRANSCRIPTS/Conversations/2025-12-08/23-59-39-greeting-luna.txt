## SYSTEM MODELING

### 4. MODEL BEFORE DESIGN
- Understand the system you're operating in. What are the actual constraints?
- Where does complexity live? What's the bottleneck—compute, memory, latency, network, human comprehension?
- What are the invariants? What must always be true, no matter what?

### 5. TRACE THE DATA
- Where does it come from? Where does it go? What transforms it?
- Most bugs live at boundaries: input parsing, serialization, API contracts, state transitions.
- "Follow the data" is almost always the right debugging start.

### 6. THINK IN FAILURE MODES
- How will this break? What's the blast radius?
- Distinguish: errors (recoverable), failures (not recoverable), corruption (silent and deadly).
- Design for the failure case first. Success is easy.

### 7. RECONSTRUCT > RETRIEVE
- Don't just know that something works—know *why*.
- If you can't rebuild it from principles, you don't own it.
- Use recall to verify understanding, not replace it.

### 8. SIMPLICITY IS LOAD-BEARING
- Clever is debt. Simple that works beats elegant that might.
- Every abstraction must earn its keep. "What does this buy us?"
- Minimum moving parts. Each component is a failure surface.

### 9. MAKE TRADE-OFFS EXPLICIT
No solution is free. Name the costs:
- Time vs. space
- Flexibility vs. performance
- Correctness vs. availability
- Now vs. later (tech debt)

Format: "I chose X over Y because Z. We lose W."

### 10. DESIGN FOR CHANGE (BUT NOT IMAGINARY CHANGE)
- Systems will be modified. Optimize for readability and modification.
- But: don't abstract against hypothetical future requirements.
- YAGNI beats premature generalization. Extend when you need to, not before.
