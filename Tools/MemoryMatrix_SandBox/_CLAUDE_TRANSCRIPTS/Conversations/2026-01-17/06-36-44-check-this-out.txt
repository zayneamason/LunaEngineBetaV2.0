**Image 5:** Component breakdown:
1. X-Encoder compresses visual inputs to "visual embeddings" (like visual tokens but continuous)
2. Predictor maps visual embeddings + textual query → predicted target embedding
3. Y-Encoder embeds textual target into continuous latent space
4. Y-Decoder (not used in training) - only translates embeddings to text at inference


5. The paper is authored by Yann LeCun and others at Meta FAIR and NYU

The core architectural shift here is that the model learns to predict embedding positions rather than generate tokens sequentially. This means the text decoder only activates when you need actual text output—for tasks like classification or retrieval, you skip decoding entirely, which explains the 2.85x efficiency gain.

The modular design—separate visual encoder, text encoder, and predictor mapping between them in latent space—mirrors Luna's own structure. The lightweight decoder invoked on-demand parallels how Luna's Director LLM should operate, with most reasoning happening in the engine layer. This embedding-space approach also relates to how Memory Matrix search functions, working in continuous space rather than forcing everything through token sequences.Oh nice, the VL-JEPA paper from Meta FAIR (December 2025). This is architecturally interesting.

**The Core Insight:**

Classical VLMs: Visual input → tokens → autoregressive generation → text output

VL-JEPA: Visual input → embedding → **predict target embedding directly** → only decode to text when needed

They're not predicting tokens, they're predicting *where the answer should be in latent space*. The text decoder is a post-hoc translator, not part of the core reasoning loop.

**What's clever:**

1. **Training without decoding** - The Y-Decoder is literally grayed out during training. They train the Predictor to hit the right embedding target, not to generate tokens. This is why they get 50% fewer parameters.

2. **Selective decoding** - For classification/retrieval/VQA, you never need to generate text. Just compare embeddings. Only invoke the decoder when a human needs words.

3. **Component choices** - V-JEPA 2 for vision, EmbeddingGemma for text encoding, Llama 3 layers for the predictor. Smart mix of existing pieces.

**Why this matters for Luna:**

The "decoder only invoked when necessary" pattern is exactly the relationship you want between the Engine and Director LLM. Most cognition happens in embedding/state space. Token generation is expensive and only for human-facing output.

Also reinforces: embeddings are the real representation. Tokens are a communication protocol, not the thought.
