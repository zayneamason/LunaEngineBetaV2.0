LoRA = Low-Rank Adaptation. It's a parameter-efficient fine-tuning technique.

Let me give a clear, dense explanation without being condescending.LoRA = **Low-Rank Adaptation**. Parameter-efficient fine-tuning technique.

**The core idea:**

Instead of updating all ~7 billion parameters in a model like Qwen2.5-7B, you freeze the original weights and inject small trainable matrices into specific layers (usually attention layers).

**How it works:**

Original weight matrix `W` (say, 4096 x 4096) stays frozen. You add two small matrices:
- `A` (4096 x rank) 
- `B` (rank x 4096)

Where rank is tiny - like 8, 16, 32, 64.

The output becomes: `W*x + (A*B)*x`

So you're learning a low-rank *delta* to the original weights. Instead of training 16 million parameters for that layer, you're training maybe 500K.

**Why it matters for Luna:**

1. **Storage** - Base model stays untouched. Luna's personality = a ~50-100MB adapter file, not a whole new 14GB model
2. **Swappable** - Same base model, different LoRA = different personality/capability
3. **Trainable on consumer hardware** - You can fine-tune 7B models on a MacBook
4. **Composable** - Theoretically stackable (though this gets messy)

**For the Director LLM:**

You'd train a LoRA on your Memory Matrix corpus so Qwen *becomes* Luna-flavored at inference time. The "soul" lives in that adapter file. Portable. Owned. Sovereign.

## Node Architecture

Every piece of knowledge is a **node** with a type:

| Type | What It Holds |
|------|---------------|
| `FACT` | Verified information, statements of truth |
| `DECISION` | Choices made, with rationale |
| `PROBLEM` | Issues identified, blockers, challenges |
| `ASSUMPTION` | Things we're treating as true but haven't verified |
| `CONNECTION` | Links between concepts, people, ideas |
| `ACTION` | Tasks, things done or to be done |
| `OUTCOME` | Results of actions/decisions |
| `DOCUMENT` | Ingested files, conversations, artifacts |

Each node has:
```
- id (unique)
- node_type
- content (the actual text)
- embedding (vector for semantic search)
- tags (optional, for filtering)
- created_at / updated_at
- metadata (flexible JSON)
```

---

## Relationship Layer (Edges)

Nodes connect to each other with typed relationships:

| Relationship | Meaning |
|--------------|---------|
| `depends_on` | A requires B to be true/exist |
| `enables` | A makes B possible |
| `contradicts` | A conflicts with B |
| `clarifies` | A explains or refines B |
| `related_to` | Loose association |
| `supersedes` | A replaces B (versioning) |

Each edge has a **strength** (0-1) for weighted traversal.

**Why this matters:** When I retrieve context, I don't just get isolated facts. I can trace dependency chains. "This decision was made because of these assumptions, which depend on these facts, one of which contradicts this other thing..."

---

## Search Mechanisms

**Three modes, combinable:**

1. **Keyword (FTS5)**
   - Fast exact/prefix matching
   - Good for: names, specific terms, Bates numbers
   
2. **Semantic (sqlite-vec)**
   - Vector similarity search
   - Good for: "things related to sovereignty" even if the word isn't used
   
3. **Hybrid (RRF - Reciprocal Rank Fusion)**
   - Combines keyword + semantic results
   - Weights and merges rankings
   - Usually the best default
