**Current efficiency: Pretty bad.**

Let's break it down:

## Storage Efficiency: ✅ GREAT
- Raw text is tiny (25-50MB for everything)
- SQLite databases are compact
- Transcripts compress well
- **Not a problem**

## Retrieval Efficiency: ⚠️ MIXED

**What works:**
- FTS5 search: Fast keyword lookup (milliseconds)
- Vector search: Semantic similarity (sub-second with good indexes)
- Lock-in scoring: O(1) per node

**What doesn't:**
- **No caching** — every query hits the database cold
- **No precomputed clusters** — clustering happens on-demand
- **Linear scan** for spreading activation (O(nodes × edges))
- **Duplicate work** — same query twice = same computation twice

## Provenance Efficiency: ❌ TERRIBLE

**To answer "where did this come from?":**
1. Find node in Memory Matrix
2. Read `conversationRef` field → "conv_042"
3. Search filesystem for transcript matching that ID
4. Linear scan through transcript to find context
5. Parse surrounding messages for full picture

That's **5 disk operations** minimum. If you want context from 10 nodes, that's **50 operations**.

## Memory Matrix + Nexus Integration: ❌ NONEXISTENT

Right now they're **completely separate**:
- Nexus tasks = static artifact prototype
- Memory Matrix = working database
- No shared timeline
- No unified query layer
- **Zero integration**

To combine them requires:
- New schema for temporal data
- Cross-database joins or denormalization
- Timeline index rebuild
- **Significant engineering**

## The Real Problem: **Query Pattern**

Current pattern:
```
User asks → Luna searches → Gets nodes → Formats response
         ↓
    (throws away context)
```

Efficient pattern:
```
User asks → Check hot cache → 
         ↓              ↓ (miss)
    (hit) Return   → Search + cache for next time
```

**We're doing zero caching.** Every query is fresh compute.

For your question "show me the full context":

- With cache: **<1ms per node** (memory lookup)

For 100 nodes = **5-10 seconds vs 500ms vs 100ms**

1. **Add minimal provenance index** — transcript_id → node_ids mapping (cheap)
2. **Cache recent queries** — 80/20 rule applies (cheap)
3. **Defer Nexus integration** — until we actually use it for real work (expensive, low ROI)
4. **Keep it simple** — premature optimization kills projects

**Current bottleneck isn't speed — it's usefulness.** Make the system *work correctly* first, then make it *work fast*.

What matters more: sub-second retrieval or actually having the right memories?
