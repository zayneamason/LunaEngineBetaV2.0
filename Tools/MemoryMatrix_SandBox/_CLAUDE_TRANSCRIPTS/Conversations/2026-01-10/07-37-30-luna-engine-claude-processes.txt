## Hub Daemon → Runtime Engine: The Transformation

### The Core Problem with Hub

| Hub Daemon (Old) | Runtime Engine (New) |
|------------------|---------------------|
| Service-based: components wait to be called | Engine-based: components live in continuous loop |
| Request → Response | Tick → Process → Update |
| Stateless handlers | Stateful actors with mailboxes |
| Events interrupt immediately | Events buffer and poll |
| **Luna responds** | **Luna *lives*** |

The fundamental insight: **No heartbeat.** The Hub is reactive — POST arrives, handler fires. No situational awareness, no abort control, no way to handle interrupt storms.

---

## The Engine Architecture

### Three Priority Paths (Variable-Rate Heartbeat)

```
HOT PATH (Interrupt-Driven)     ← Voice partials, barge-in, TTS streaming
    ↓                              As fast as events arrive
COGNITIVE PATH (~1-2 Hz)        ← Director heartbeat, "Should I prefetch?"
    ↓                              Every 500ms-1s  
REFLECTIVE PATH (Minutes)       ← Pruning, summarization, consolidation
                                   Every 5-30 minutes
```

This is the key insight from Gemini: a fixed-frequency tick (like Unity's 60fps) makes no sense for an LLM that takes 500ms-3s to think.

### Buffered Input Model

Events land in an async buffer. Engine polls at heartbeat. This gives:
- **Situational awareness** — sees all pending input before deciding
- **Abort control** — can cancel inference if user interrupts
- **Priority ordering** — interrupts first, then finals, then partials
- **No interrupt storms** — buffer absorbs bursts

### Actor Model (Erlang/Akka Style)

Six actors, each with isolated state and mailbox:

| Actor | Owns | Failure Mode |
|-------|------|--------------|
| Director | LLM, state machine | Returns graceful error |
| Matrix | SQLite/vectors/graph | Returns empty results |
| Voice | STT/TTS/audio | Silence, retry |
| Scribe | Extraction, chunking | Queue for later |
| Librarian | Filing, edges, pruning | Queue for later |
| Oven | Claude delegation | Timeout, local fallback |

**The killer feature:** If Matrix crashes during a weird search, Director doesn't die. It receives empty results and says "I'm having trouble remembering that right now" instead of the whole engine freezing.

### State Serialization

```
on_snapshot() → serialize actor states → sleep
on_restore() → deserialize → resume exactly where you left off
```

## So what does the engine actually DO?

**Core loop, every interaction:**

```python
async def process_message(user_input: str) -> str:
    # 1. CONTEXT ASSEMBLY
    identity = persona_core.get_current_identity()      # Who is Luna right now?
    memories = memory_matrix.retrieve(user_input)       # What's relevant?
    state = consciousness.get_current_state()           # Attention, mood, personality
    tools = mcp_registry.get_available_tools()          # What can Luna do?
    history = session.get_recent_turns()                # What just happened?
    
    # 2. BUILD PROMPT
    prompt = assemble_prompt(
        system=identity,
        memories=memories,
        state=state,
        tools=tools,
        history=history,
        user_message=user_input
    )
    
    # 3. INFERENCE (the GPU call)
    response = await llm.generate(prompt)  # Claude API or local Director
    
    # 4. TOOL LOOP (if needed)
    while response.has_tool_calls():
        results = await execute_tools(response.tool_calls)
        response = await llm.continue_with(results)
    
    # 5. POST-PROCESSING
    session.add_turn(user_input, response.text)         # Store conversation
    consciousness.update(user_input, response.text)     # Update state
    await scribe.extract(user_input, response.text)     # Ben extracts
    # Dude files async in background
    
    # 6. OUTPUT
    return response.text
```

That's the engine. Everything except step 3 is us.
