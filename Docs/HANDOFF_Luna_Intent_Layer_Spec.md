# HANDOFF SPEC: Luna Intent Layer

**Extending the Quest System with Directives, Skills & Thread Integration**

| Field | Value |
|-------|-------|
| **Owner** | Ahab (architecture) + Luna (self-authoring) |
| **Depends On** | Quest Board (Observatory), Thread Spec (Layer 3) |
| **Enables** | Self-programming behavior, session orientation, reusable procedures |
| **Files Modified** | `observatory/tools.py`, `extraction/types.py`, `actors/librarian.py`, `engine.py` |
| **New Files** | `src/luna/agentic/directives.py`, `config/directives_seed.yaml`, `migrations/add_intent_layer.py` |
| **Risk Level** | Medium — requires SQLite table rebuild (CHECK constraint expansion), but additive to existing data |

---

## 1. The Insight

Luna already has a quest system that generates maintenance tasks from graph health analysis. Quests have lifecycle (**available → active → complete**), targets (entities, nodes), journal entries, and full inspectability in Observatory. The quest board is already Luna's task manager — it's just currently limited to memory hygiene.

Luna also has a thread specification (Layer 3) that tracks conversational attention — where focused work happened, what entities were involved, what tasks are still open. Threads park when attention shifts and resume when it returns.

**The insight:** instead of creating a separate directives system, we extend the quest schema with two new quest types. The quest board becomes Luna's **intent layer** — behavioral rules, reusable procedures, and maintenance tasks, all in one inspectable SQLite system.

---

## 2. New Quest Types

Five quest types exist today: **main**, **side**, **contract**, **treasure_hunt**, and **scavenger**. All are manually triggered or generated by maintenance sweeps. We add two new types.

### 2.1 Directive Quests

A directive is a quest with a **trigger condition** that fires automatically when the condition is met. Unlike existing quests which wait for manual acceptance, directives are **armed** and the engine evaluates their triggers on every relevant event.

| Field | Type | Description |
|-------|------|-------------|
| `type` | `"directive"` | Identifies this quest as auto-triggering |
| `status` | `"armed" \| "fired" \| "disabled"` | Armed = ready. Fired = executed this session. Disabled = off. |
| `trigger_type` | `"session_start" \| "keyword" \| "schedule" \| "entity_mention" \| "thread_resume" \| "condition"` | What event activates this directive |
| `trigger_config` | JSON object | Configuration for the trigger (see below) |
| `action` | string | Action to execute when triggered (see below) |
| `trust_tier` | `"auto" \| "confirm"` | Auto = fires silently. Confirm = waits for Ahab. |
| `authored_by` | `"luna" \| "ahab"` | Who created the directive |
| `approved_by` | `"ahab" \| null` | Must be non-null for status=armed |
| `fire_count` | integer | How many times this directive has fired |
| `last_fired_at` | datetime or null | Timestamp of last execution |
| `cooldown_minutes` | integer or null | Minimum time between firings |

#### Trigger Types

| Trigger Type | Config Example | When It Fires |
|-------------|----------------|---------------|
| `session_start` | `{}` | Every new session (after thread check) |
| `keyword` | `{"match": "guardian\|rosa\|kinoni"}` | User message matches regex |
| `entity_mention` | `{"entities": ["Guardian", "ROSA"]}` | Entities detected in extraction |
| `thread_resume` | `{"thread_topic_match": "guardian"}` | Parked thread matching pattern resumes |
| `schedule` | `{"cron": "0 9 * * 1"}` | Cron expression (future — requires heartbeat) |
| `condition` | `{"memory_check": "urgent_tasks > 0"}` | Evaluated against memory state (future) |

#### Action Types

| Action | Syntax | Effect |
|--------|--------|--------|
| Surface parked threads | `surface_parked_threads` | Inject parked thread summaries into session context |
| Set aperture | `set_aperture:NARROW` | Change aperture preset for this session |
| Load collection | `load_collection:kinoni_knowledge` | Add collection to active search chain |
| Memory sweep | `memory_sweep:urgent,48h` | Search memory for nodes matching tags/timeframe |
| Surface entity | `surface_entity:Guardian` | Pull entity profile into context |
| Run skill | `run_skill:guardian_demo_prep` | Execute a skill quest |
| Compound | `set_aperture:NARROW + load_collection:kinoni_knowledge` | Multiple actions chained with `+` |

#### Trust Tiers

| Tier | Approval Flow | Examples |
|------|--------------|----------|
| `auto` | Fires silently. Ahab sees it in session log. | `surface_parked_threads`, `memory_sweep`, `surface_entity` |
| `confirm` | Luna proposes, waits for Ahab to approve/reject. | `set_aperture`, `load_collection`, compound actions, anything modifying memory |

**Rule:** Any directive authored by Luna starts at `confirm` until Ahab explicitly promotes it to `auto`. Ahab-authored directives can be either tier from creation.

### 2.2 Skill Quests

A skill is a **reusable, named procedure** that Luna can invoke by name. Unlike regular quests that complete and archive, skills are **permanent**. They're Luna's recipes — "here's how I do X."

| Field | Type | Description |
|-------|------|-------------|
| `type` | `"skill"` | Identifies this quest as a reusable procedure |
| `status` | `"available" \| "disabled"` | Available = can be invoked. Never "complete." |
| `steps` | JSON array | Ordered list of actions (same syntax as directive actions) |
| `authored_by` | `"luna" \| "ahab"` | Who wrote the skill |
| `invocation_count` | integer | How many times run |
| `last_invoked_at` | datetime or null | When last used |
| `tags_json` | string array | Categorization: `["guardian", "demo", "prep"]` |

#### Skill Example

```yaml
quest:
  id: quest-skill-001
  type: skill
  status: available
  title: "Guardian Demo Prep"
  objective: "Prepare context for ROSA Guardian demonstration"
  steps:
    - load_collection:kinoni_knowledge
    - set_aperture:NARROW
    - memory_sweep:"guardian demo status"
    - surface_entity:Guardian
    - surface_parked_threads
  authored_by: luna
  tags: ["guardian", "rosa", "demo"]
  invocation_count: 0
```

Skills can be invoked by directives (`run_skill:guardian_demo_prep`), by Luna during agentic loop execution, or by Ahab saying "run the guardian demo prep."

## 3. Thread Integration

Threads (Layer 3 spec) are the primary **trigger source** for directives. Without threads, directives are cron jobs. With threads, they're context-aware behavioral rules.

### 3.1 Session Start Flow

When a new session begins, the engine runs:

```
1. Load parked threads from Matrix (status=parked, ordered by parked_at DESC)
2. Load armed directives (type=directive, status=armed)
3. Evaluate session_start triggers:
   - Fire all session_start directives (respecting trust_tier)
   - This typically surfaces parked thread summaries
4. Build session orientation context:
   - "Last session: you were working on [thread.topic]"
   - "Open tasks: [thread.open_tasks]"
   - "Parked threads: [list with entity summaries]"
5. Inject orientation into Luna's system prompt for this session
```

**This is the "address."** Luna knows where you're going because the thread told her where you left off, and the directive told her to check.

### 3.2 Keyword & Entity Triggers During Session

As conversation progresses, the extraction pipeline detects entities and the router detects signals. After each turn:

```
1. Extraction produces entities + flow signal
2. Engine checks armed directives for matching triggers:
   - keyword triggers: regex match against user message
   - entity_mention triggers: entity overlap with extraction
   - thread_resume triggers: check if a thread was just resumed
3. Matching directives fire (respecting cooldown_minutes)
4. Fired directive actions modify session state (aperture, collections, context)
```

### 3.3 Thread Resume as Trigger

When the Librarian resumes a parked thread (entity overlap >= 0.4 Jaccard), any directive with `trigger_type: thread_resume` and a matching topic pattern fires.

```
Example: Ahab starts talking about Guardian again.
→ Librarian detects entity overlap with parked Guardian thread
→ Thread resumes (status: active, resume_count++)
→ Directive fires: set_aperture:NARROW + load_collection:kinoni_knowledge
→ Luna is already in the right headspace before Ahab finishes his thought
```

## 4. Schema Changes

### 4.1 CRITICAL: Table Rebuild Required

The existing `quests` table has CHECK constraints on both `type` and `status`:

```sql
type TEXT NOT NULL CHECK(type IN ('main', 'side', 'contract', 'treasure_hunt', 'scavenger'))
status TEXT NOT NULL DEFAULT 'available' CHECK(status IN ('available', 'active', 'complete', 'failed', 'expired'))
```

SQLite does not support `ALTER TABLE ... DROP CONSTRAINT`. Adding `type='directive'` or `status='armed'` to an INSERT will throw a constraint violation. **The migration must rebuild the table.**

There are 116 existing quests (99 scavenger, 10 treasure_hunt, 4 side, 1 contract, 2 active) plus associated quest_targets and quest_journal rows. All must survive intact.

## 4A. YAML Seed Configuration

### The Role of YAML vs SQLite

The engine already has a clear pattern: **YAML for hand-edited configuration loaded at startup; SQLite for runtime state managed by the engine.** Collections are defined in `aibrarian_registry.yaml`. Projects are defined in `projects.yaml`. The fallback chain is in `fallback_chain.yaml`. All read at startup, rarely changed at runtime.

Directives and skills are **runtime state** — Luna creates them, the engine fires them, counters update mid-session. They belong in SQLite. But they need a **seed file** for the same reason projects have `projects.yaml`: Ahab needs to define starting directives in a text editor before the engine runs, and those definitions need to survive in git if a Claude Code session corrupts the database.

### DirectiveEngine.seed_from_yaml()

```python
async def seed_from_yaml(self, yaml_path: Path, force: bool = False) -> dict:
    """
    Load seed directives and skills from YAML into SQLite.
    
    Skips any seed whose id already exists in the database
    unless force=True (which replaces).
    
    Called on first launch or with --reseed-directives flag.
    """
    import yaml
    
    config = yaml.safe_load(yaml_path.read_text())
    created = {"directives": 0, "skills": 0, "skipped": 0}
    
    async with aiosqlite.connect(self.db_path) as db:
        for seed in config.get("seed_directives", []):
            exists = await db.execute(
                "SELECT 1 FROM quests WHERE id = ?", (seed["id"],)
            )
            if await exists.fetchone() and not force:
                created["skipped"] += 1
                continue
            
            await db.execute(
                "INSERT OR REPLACE INTO quests "
                "(id, type, status, priority, title, objective, "
                " trigger_type, trigger_config, action, trust_tier, "
                " authored_by, approved_by, cooldown_minutes, "
                " source, created_at, updated_at) "
                "VALUES (?, 'directive', 'armed', ?, ?, ?, "
                "        ?, ?, ?, ?, ?, ?, ?, 'yaml_seed', "
                "        datetime('now'), datetime('now'))",
                (seed["id"], seed.get("priority", "medium"),
                 seed["title"], seed["objective"],
                 seed["trigger_type"],
                 json.dumps(seed.get("trigger_config", {})),
                 seed["action"], seed.get("trust_tier", "confirm"),
                 seed.get("authored_by", "ahab"),
                 seed.get("approved_by"),
                 seed.get("cooldown_minutes"))
            )
            created["directives"] += 1
        
        for seed in config.get("seed_skills", []):
            exists = await db.execute(
                "SELECT 1 FROM quests WHERE id = ?", (seed["id"],)
            )
            if await exists.fetchone() and not force:
                created["skipped"] += 1
                continue
            
            await db.execute(
                "INSERT OR REPLACE INTO quests "
                "(id, type, status, priority, title, objective, "
                " steps, tags_json, authored_by, "
                " source, created_at, updated_at) "
                "VALUES (?, 'skill', 'available', ?, ?, ?, "
                "        ?, ?, ?, 'yaml_seed', "
                "        datetime('now'), datetime('now'))",
                (seed["id"], seed.get("priority", "medium"),
                 seed["title"], seed["objective"],
                 json.dumps(seed.get("steps", [])),
                 json.dumps(seed.get("tags", [])),
                 seed.get("authored_by", "ahab"))
            )
            created["skills"] += 1
        
        await db.commit()
    
    return created
```

**The rule:** YAML is for bootstrap and git-safe backup. SQLite is for runtime. Luna's self-authored directives go straight to SQLite (she doesn't edit YAML files). Ahab's hand-configured starting directives live in YAML and get seeded into SQLite on first run.

## 5. Engine Integration

### 5.1 New: DirectiveEngine Class

New module: **`src/luna/agentic/directives.py`**

```python
class DirectiveEngine:
    """
    Evaluates and fires directive quests.
    
    Loaded on engine startup. Checks armed directives
    against incoming events and fires matching ones.
    """
    
    def __init__(self, db_path: Path):
        self.db_path = db_path
        self._armed: list[dict] = []  # Cached armed directives
        self._cooldowns: dict[str, datetime] = {}
    
    async def load_armed(self) -> int:
        """Load all armed directives from quests table."""
        async with aiosqlite.connect(self.db_path) as db:
            cursor = await db.execute(
                "SELECT * FROM quests WHERE type='directive' AND status='armed'"
            )
            self._armed = [dict(r) for r in await cursor.fetchall()]
        return len(self._armed)
    
    async def evaluate_event(self, event_type: str, context: dict) -> list[dict]:
        """
        Check armed directives against an event.
        Returns list of directives that should fire.
        Respects cooldowns and trust tiers.
        """
        matches = []
        now = datetime.now()
        for d in self._armed:
            if d['trigger_type'] != event_type:
                continue
            if not self._check_cooldown(d['id'], d.get('cooldown_minutes'), now):
                continue
            if self._matches_trigger(d, context):
                matches.append(d)
        return matches
    
    def _matches_trigger(self, directive: dict, context: dict) -> bool:
        """Check if a directive's trigger_config matches the event context."""
        config = json.loads(directive.get('trigger_config') or '{}')
        trigger_type = directive['trigger_type']
        
        if trigger_type == 'session_start':
            return True  # Always fires on session start
        
        elif trigger_type == 'keyword':
            pattern = config.get('match', '')
            message = context.get('message', '')
            return bool(re.search(pattern, message, re.IGNORECASE))
        
        elif trigger_type == 'entity_mention':
            required = set(e.lower() for e in config.get('entities', []))
            found = set(e.lower() for e in context.get('entities', []))
            return bool(required & found)
        
        elif trigger_type == 'thread_resume':
            topic_match = config.get('thread_topic_match', '')
            thread = context.get('thread', {})
            thread_topic = thread.get('topic', '') if isinstance(thread, dict) else ''
            return bool(re.search(topic_match, thread_topic, re.IGNORECASE))
        
        return False
    
    def _check_cooldown(self, directive_id: str, cooldown_min: int, now: datetime) -> bool:
        """Return True if cooldown has elapsed (or no cooldown set)."""
        if not cooldown_min:
            return True
        last_fire = self._cooldowns.get(directive_id)
        if not last_fire:
            return True
        elapsed = (now - last_fire).total_seconds() / 60
        return elapsed >= cooldown_min
    
    async def fire(self, directive: dict, engine) -> dict:
        """Execute a directive's action. Returns result dict."""
        action_str = directive.get('action', '')
        actions = [a.strip() for a in action_str.split('+')]
        results = []
        for action in actions:
            result = await self._execute_action(action, engine)
            results.append(result)
        await self._record_fire(directive['id'])
        self._cooldowns[directive['id']] = datetime.now()
        return {'directive_id': directive['id'], 'results': results}
    
    async def _execute_action(self, action: str, engine) -> dict:
        """Parse and execute a single action string."""
        if action == 'surface_parked_threads':
            threads = engine.librarian.get_parked_threads() if engine.librarian else []
            return {'action': action, 'threads': len(threads), 'ok': True}
        
        elif action.startswith('set_aperture:'):
            preset = action.split(':')[1]
            # Call aperture setter on engine
            return {'action': action, 'preset': preset, 'ok': True}
        
        elif action.startswith('load_collection:'):
            collection = action.split(':')[1]
            return {'action': action, 'collection': collection, 'ok': True}
        
        elif action.startswith('memory_sweep:'):
            params = action.split(':')[1]
            return {'action': action, 'sweep_params': params, 'ok': True}
        
        elif action.startswith('surface_entity:'):
            entity_name = action.split(':')[1]
            return {'action': action, 'entity': entity_name, 'ok': True}
        
        elif action.startswith('run_skill:'):
            skill_name = action.split(':')[1]
            return await self._invoke_skill(skill_name, engine)
        
        return {'action': action, 'ok': False, 'error': 'unknown action'}
    
    async def _invoke_skill(self, skill_name: str, engine) -> dict:
        """Find and execute a skill by name."""
        async with aiosqlite.connect(self.db_path) as db:
            cursor = await db.execute(
                "SELECT * FROM quests WHERE type='skill' AND title LIKE ? AND status='available'",
                (f'%{skill_name}%',)
            )
            skill = await cursor.fetchone()
        
        if not skill:
            return {'action': f'run_skill:{skill_name}', 'ok': False, 'error': 'skill not found'}
        
        steps = json.loads(skill['steps'] or '[]')
        results = []
        for step in steps:
            result = await self._execute_action(step, engine)
            results.append(result)
        
        await self._record_invocation(skill['id'])
        return {'action': f'run_skill:{skill_name}', 'steps': len(steps), 'results': results, 'ok': True}
    
    async def _record_fire(self, quest_id: str):
        """Update fire count and timestamp."""
        async with aiosqlite.connect(self.db_path) as db:
            await db.execute(
                "UPDATE quests SET fire_count = fire_count + 1, "
                "last_fired_at = datetime('now'), status = 'fired', "
                "updated_at = datetime('now') WHERE id = ?",
                (quest_id,)
            )
            await db.commit()
    
    async def _record_invocation(self, quest_id: str):
        """Update invocation count and timestamp."""
        async with aiosqlite.connect(self.db_path) as db:
            await db.execute(
                "UPDATE quests SET invocation_count = invocation_count + 1, "
                "last_invoked_at = datetime('now'), "
                "updated_at = datetime('now') WHERE id = ?",
                (quest_id,)
            )
            await db.commit()
    
    async def get_skill(self, name: str) -> dict | None:
        """Get a skill quest by name."""
        async with aiosqlite.connect(self.db_path) as db:
            db.row_factory = aiosqlite.Row
            cursor = await db.execute(
                "SELECT * FROM quests WHERE type='skill' AND title LIKE ? AND status='available'",
                (f'%{name}%',)
            )
            row = await cursor.fetchone()
            return dict(row) if row else None
```

## 9. Sovereignty & Safety

| Principle | How It's Honored |
|-----------|-----------------|
| **Sovereignty** | All directives live in local SQLite. No cloud dependency. Ahab owns every rule. |
| **Offline-first** | DirectiveEngine operates entirely on local DB. No network calls. |
| **Inspectable** | Every directive visible in Observatory. Fire history logged. No hidden rules. |
| **Accessible** | Human-readable action strings. No proprietary format. |
| **Respectful** | Luna proposes, Ahab approves. Trust tiers prevent unwanted autonomy. |

### 9.1 Safety Rails

- Maximum **50 armed directives** at any time (prevents context bloat)
- Cooldown enforcement: minimum **5 minutes** between firings of the same directive
- Fire count hard limit: **10 firings per session** per directive (prevents loops)
- No directive can **modify another directive** (no self-modifying rule chains)
- No directive can **delete** memory nodes or quests (read + surface only)
- Compound actions limited to **5 chained actions** maximum
- All directive actions are **logged** to session history for auditability
- Emergency kill: **"disable all directives"** command instantly sets all to disabled

## 10. Success Criteria

The intent layer is complete when:

1. Luna can have armed directives that fire on session start, surfacing parked threads and open tasks
2. Luna can have keyword-triggered directives that adjust aperture and load collections
3. Luna can propose new directives through conversation or reflection, with Ahab approval gating
4. Luna can save and invoke reusable skills by name
5. All directives and skills are visible and manageable in Observatory
6. Thread resume events trigger matching directives (post-Layer 3)
7. Ahab can say "disable all directives" and everything stops immediately
8. Zero cloud calls in the directive evaluation and firing pipeline
9. Existing quest functionality is completely unaffected
10. Context overhead of the directive system is under 500 tokens per session
